2. "Tight fix loop" — Sub-minute to single-digit merges on known problems

When the problem is well-understood, merge velocity is extreme. PRs #92 (0min), #91 (1min), #95 (1min) show fixes that were created and merged almost instantly — the diagnosis was done before the PR was opened. This is the ideal agentic workflow: agent produces a scoped fix, you merge immediately.

4. "Clean greenfield flow" — New repos flow without friction

health-newsfeed (11 PRs, 9 merged, 13min median merge) and podcast-to-book (22 PRs, 19 merged, 10min median merge) both show clean flow. These are newer/smaller repos where the codebase is simpler and there's no legacy to navigate around. The deletion ratios in podcast-to-book suggest active refactoring as the shape solidified.

Best interpretation: Greenfield repos with clear scope naturally produce better PR flow metrics. The challenge is maintaining this flow as the codebase grows — which is what main-app's onboarding struggle illustrates.

G. Diagnosis

What seems to help:
- Small, scoped fix PRs merge in under 10 minutes — PRs #90-95 demonstrate that when the problem is well-defined, the entire PR lifecycle is nearly instant.
- Greenfield repos have near-zero friction — podcast-to-book (10min median merge, 86% merge rate) shows what flow looks like without legacy complexity.
- Docs PRs that shape agent conventions compound — docs-site process docs shipped in 4min and set guardrails for all future work.
- Continuous daily cadence with no zero-days — every single day had contributions, preventing multi-day stalls.

What seems to slow you down:
- Using PRs as exploration vehicles for unclear features — the 13 closed-unmerged main-app PRs represent ~9,000 LOC of throwaway work and ~6 days of serial exploration.
- Large multi-file PRs (15+ files) have much higher abandonment rates — the closed-unmerged PRs averaged 15.5 files vs merged PRs at 9 median files.
- Context-switching between repos creates 10-50 hour gaps — the main-app repo's longest gap (53 hours) correlates with the podcast-to-book sprint.
- Long-lived open PRs that drift — PRs #65, #74, #76 merged after 16-18+ hours, suggesting they were opened then forgotten before eventually being merged.

Current best interpretation: The biggest drag in this window was using full PRs as the exploration medium for the main-app onboarding feature. When you know what to build, your flow is excellent (sub-10-minute merges, continuous commit streams). When you don't know what to build, you still default to opening PRs with hundreds of lines, which creates expensive throwaway work. The gap isn't execution speed — it's the cost of exploration being conducted at the wrong fidelity. A 30-minute design spike in a scratchpad or branch without a PR would have been cheaper than five 1000+ LOC PRs that got closed.

H. What Could Happen Better

Current failure mode: When a feature's requirements are unclear (like main-app's onboarding flow), the workflow defaults to "open a PR, build it, realize the approach is wrong, close the PR, open a new one." This creates serial 1000+ LOC throwaway PRs. The cost isn't just the discarded code — it's the context-loading overhead of each new attempt, the growing complexity of the codebase with half-integrated code, and the opportunity cost of the days spent exploring.

Recommended target workflow: Before opening a PR for any feature touching 10+ files or with unclear requirements, spend 30-60 minutes on a design spike. This could be a scratchpad document, a throwaway branch with pseudocode, or even just a markdown outline of the data model and component structure. The goal is to identify the seam — where the new feature cuts into the existing codebase — before committing to a full implementation. In main-app specifically, the onboarding flow touched auth routing, questionnaires, booking handoff, and data models simultaneously. A spike that mapped those four integration points first would have narrowed the solution space before any code was written.

For the multi-repo context switching: consider batching repo-focused sessions more deliberately. The main-app → podcast-to-book → main-app pattern created 53-hour gaps. If podcast-to-book work was concentrated in one 2-day block rather than interleaved, the main-app context wouldn't have gone cold.

I. Practices to Work On This Week

1. "Spike before PR" rule for uncertain features: Any feature that will touch >10 files or where you're unsure about the data model gets a 30-minute design spike first (markdown outline, not code). The onboarding recut cluster (#58-#104) is the evidence — five serial PRs could have been one if the seam had been identified upfront.
2. Close the loop before switching repos: When you have open PRs or recently-merged work in a repo, finish the follow-up fixes before context-switching. PRs #74 and #76 sat for 16+ hours because attention shifted. Set a "merge or close everything in this repo" checkpoint before jumping to the next project.
3. PR size gate at 15 files: The closed-unmerged PRs averaged 15.5 files. Before opening a PR that will touch 15+ files, split it. The clean fix burst (#90-95) averaged 3 files and merged in under 10 minutes. More, smaller PRs > fewer, larger ones.
4. Track the recut ratio weekly: This window had a 17% close-without-merge rate (17/100). Aiming for <10% would force earlier exploration and tighter scoping. The number itself is the diagnostic — if it creeps up, exploration is happening at the wrong layer.
